# 代码修改：小步快跑、限制范围与验收标准

这一章讲的是怎么让 Claude Code 真正动手改代码：先出计划、限定范围、保持最小 diff，最后用明确的验收标准收尾。

## 使用场景

修一个小 bug、补一个小功能、调整某个组件的行为、改接口字段、更新文档、加一条校验逻辑——这些场景都适合用本章的方法。核心原则是一次只做一件事，先让 Claude Code 说清楚打算怎么改，再放手让它去编辑。

关于权限模式：`--permission-mode` 支持 6 个值——`default`（全部需确认）、`acceptEdits`（自动接受文件编辑，命令仍需确认）、`plan`（只读探索和规划）、`auto`（自动选择）、`dontAsk`（使用 settings 中的预授权规则）、`bypassPermissions`（全部自动放行，仅限 CI/CD 等受控环境）。改代码时，`acceptEdits` 适合你已经信任修改方向、只想控制命令执行的情况。

⚠️ 改代码时尤其注意：别让 Claude Code 自动提交代码或执行会动数据库、动生产配置的命令，除非你明确要求。

普通终端命令：

```bash
cd /path/to/project
claude
```

Claude Code 对话输入：

```markdown
我要修复一个小问题：保存用户资料时，空昵称应该被拒绝。
请先只读分析相关文件，给出修改计划。不要先改代码。
限制范围：优先只改 src/user/profile.ts 和对应测试文件。
验收标准：新增或更新测试；相关测试通过；diff 保持最小。
```

## 推荐提示词模板

Claude Code 对话输入：

```markdown
请按"小步修改"流程工作。

目标：
[写清楚要改的行为]

限制范围：
- 优先只修改：[文件或目录]
- 不要修改：[明确禁止的文件或目录]
- 不要做整库格式化、批量重构、依赖升级或无关清理

流程：
1. 先只读分析相关代码
2. 给出计划和预计会改的文件
3. 等我确认后再编辑
4. 修改后展示关键 diff 摘要
5. 运行最小必要验证

验收标准：
- [行为标准]
- [测试标准]
- [兼容性标准]
- [文档或错误信息标准，如需要]

高风险动作必须先问我：删除文件、安装依赖、改权限、提交代码、联网、读取密钥。
```

确认计划后再输入：

Claude Code 对话输入：

```markdown
计划可以。请只修改你刚才列出的文件，保持最小 diff。修改完成后运行你建议的最小测试命令。
```

## 错误示范

Claude Code 对话输入：

```markdown
这个功能有点问题，你看着改，顺便优化一下代码质量。
```

**问题在哪：** 目标不清楚，边界也不清楚。"顺便优化"往往会引入一堆和问题无关的改动。

Claude Code 对话输入：

```markdown
把用户模块都重写成更好的写法。
```

**问题在哪：** 没有验收标准，也没说哪些行为不能变，风险比普通修复大得多。

普通终端命令：

```bash
claude -p "修复所有 lint 和 test 问题，并提交代码"
```

**问题在哪：** "修复所有问题"范围太大，提交代码又是高风险操作。而且 `-p` 是非交互模式，跑完就退出了，根本没机会审查 diff。

## 正确示范

Claude Code 对话输入：

```markdown
请修复订单金额为 0 时仍可提交的问题。
限制范围：只考虑 src/order/submit.ts 和 tests/order/submit.test.ts。
请先解释当前逻辑为什么放过了 0，再给出最小修改方案，等我确认。
```

Claude Code 对话输入：

```markdown
请实现方案 A。不要改公共接口，不要格式化整文件。完成后运行订单提交相关测试即可。
```

普通终端命令：

```bash
npm test -- tests/order/submit.test.ts
```

如果测试命令可能不存在，先让 Claude Code 识别项目脚本：

Claude Code 对话输入：

```markdown
请先查看 package.json 和测试目录，判断应该运行哪条最小测试命令。不要安装依赖。
```

## 练习任务

1. 选一个很小的修改——比如补个字段校验或改个文案。
2. 写出限制范围和验收标准。
3. 让 Claude Code 先读代码、给计划，不要直接改。
4. 确认计划后让它修改。
5. 让它运行最小测试，或说明为什么跑不了。
6. 审查 diff，要求它解释每个改动对应哪条验收标准。

进阶练习：故意加一个"不许动某个文件"的限制，看看 Claude Code 是不是真的遵守。

## 验收标准

- 修改前有计划，计划中列出了预计文件和验证方式。
- 实际修改文件没有超出你给定的范围。
- diff 里只有实现目标所需的内容，没有顺手重构或整库格式化。
- 至少跑了最小必要验证，或明确说明了跑不了的原因。
- 高风险动作都经过了你的确认。
- 最终总结能对应到最初的验收标准。
