# 调试专题：报错分析、复现问题与定位根因

这一章讲的是怎么让 Claude Code 按"复现 → 收集证据 → 定位根因 → 最小修复 → 验证"的顺序调试，而不是看到报错就盲改一通。

## 使用场景

测试失败、运行时报错、构建挂了、接口返回异常、页面行为不对、偶发问题、线上日志分析——这些都适合用本章的方法。调试最重要的一步是把问题变得可复现：把错误命令、输入数据、错误栈、期望行为和实际行为一起交给 Claude Code。

⚠️ 调试时别让 Claude Code 直接删数据、清缓存目录、重装依赖、连生产环境。需要跑命令时，先确认命令会不会写数据库、调外部服务或改文件。

普通终端命令：

```bash
npm test -- tests/login.test.ts
```

Claude Code 对话输入：

```markdown
这个命令失败了。请先分析错误，不要修改代码。

复现命令：`npm test -- tests/login.test.ts`

错误现象：
[粘贴关键错误栈，不要粘贴密钥]

期望行为：
[写明应该发生什么]

请输出：复现步骤、最可能根因、需要查看的文件、最小修复方案和验证命令。
```

## 推荐提示词模板

Claude Code 对话输入：

```markdown
请按调试流程处理这个问题。

问题描述：
[现象]

复现方式：
普通终端命令：[命令]

错误信息：
[错误栈或日志摘要]

约束：
- 先不要改代码
- 不要删除文件、改权限、安装依赖、联网、提交代码或读取密钥
- 如果需要运行命令，先说明命令目的、副作用和预计耗时

请先输出：
1. 错误栈逐层解释
2. 已知事实和假设
3. 需要检查的文件
4. 根因定位计划
5. 最小修复方案
6. 验证命令
```

确认定位后再输入：

Claude Code 对话输入：

```markdown
请按最小修复方案修改。只改与根因直接相关的文件。不要顺手重构。修改后运行复现命令验证。
```

## 错误示范

Claude Code 对话输入：

```markdown
报错了，帮我修一下。
```

**问题在哪：** 没给命令、没给错误栈、没说期望行为，Claude Code 只能靠猜。

Claude Code 对话输入：

```markdown
测试挂了，你把所有测试都改到通过。
```

**问题在哪：** 这样做很可能是把测试的期望值改成适应错误行为，治标不治本，真正的 bug 被掩盖了。

普通终端命令：

```bash
rm -rf node_modules && npm install && npm test
```

**问题在哪：** 删除和重装依赖都是高风险操作，可能掩盖问题或引入新变量。先确认有没有必要再说。

## 正确示范

Claude Code 对话输入：

```markdown
请分析这个失败，不要修改代码。

普通终端命令：pytest tests/test_checkout.py::test_coupon_expired

错误摘要：
AssertionError: expected 400, got 200

期望行为：过期优惠券不能被使用，接口应返回 400。
请追踪优惠券校验逻辑，找出为什么过期时间没有生效。
```

Claude Code 对话输入：

```markdown
根因确认后，请只做最小修复，并补一个回归测试证明过期优惠券会被拒绝。不要修改其它优惠券逻辑。
```

普通终端命令：

```bash
pytest tests/test_checkout.py::test_coupon_expired
```

## 练习任务

1. 找一个能稳定复现的失败测试，或者自己构造一个小 bug。
2. 把复现命令、错误摘要、期望行为写给 Claude Code。
3. 要求它先解释错误栈，不要改代码。
4. 让它列出根因定位计划和需要查看的文件。
5. 确认后让它做最小修复。
6. 跑同一条复现命令，并让 Claude Code 解释为什么现在通过了。

进阶练习：让 Claude Code 区分"直接原因"和"根因"，并判断是否需要补回归测试。

## 验收标准

- 有明确的复现步骤和复现命令。
- Claude Code 在改代码之前解释了错误栈、列出了事实和假设、给出了定位计划。
- 修复直接对应根因，没有大范围重写或无关格式化。
- 原来失败的用例通过了，并且尽量补了回归测试。
- 验证命令和结果记录在了最终总结中。
- 没有执行未经确认的高风险操作。
